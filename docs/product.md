# Product

## 1. 定位

当前控制面不是企业后台，也不是多租户平台，而是一个自用的 MCP 聚合控制台。

它的目标只有三个：

1. 接入多个 MCP 来源
2. 统一管理最终暴露的工具
3. 托管部分 MCP 来源的运行

一句话定义：

> 一个面向自用的 MCP Source / Tool / Hosted 三层控制台。

## 2. 产品边界

### 要做的

首版只做三件事：

- MCP 来源聚合
- 工具暴露层编排
- 部分来源托管运行

### 不做的

当前阶段不做：

- workspace 体系作为产品中心
- 版本快照与回滚体系
- 多用户与权限系统
- 审计日志
- 组织、团队、计费
- 重型 dashboard
- 通用 PaaS 化托管平台
- 统一 HTTP MCP 中继平台

## 3. 核心心智模型

控制台围绕三层对象展开：

### Source

一个 MCP 来源，可能是：

- 远程 HTTP
- 本地 stdio
- 托管 npm
- 托管单文件

### Tool Exposure

一个来源中的工具如何被最终暴露：

- 启用或隐藏
- 改名
- 处理命名冲突

### Hosted Runtime

托管来源的运行态：

- 启动
- 停止
- 重启
- 日志查看

## 4. 关键用户任务

### 任务 A：接一个新来源

1. 新增 Source
2. 配置来源参数
3. 测试连接或刷新能力
4. 自动发现 tools/resources/prompts

### 任务 B：决定暴露哪些工具

1. 打开 Tools 页面
2. 查看所有来源汇总后的工具列表
3. 关闭不想暴露的工具
4. 处理同名冲突
5. 输出最终暴露结果

### 任务 C：托管一个 MCP 服务

1. 新增 hosted-npm 或 hosted-single-file 来源
2. 配置启动方式
3. 启动服务
4. 查看状态和日志

## 5. 信息架构

控制台只保留三个主页面：

- `/sources`
- `/tools`
- `/hosted`

如无必要，不继续增加后台式页面。

## 6. 页面职责

### Sources

负责来源管理：

- 新增
- 编辑
- 启用/停用
- 刷新能力
- 删除

### Tools

负责最终暴露层：

- 查看所有来源的工具汇总
- 开关工具暴露
- 修改暴露名
- 处理冲突
- 查看最终暴露结果

### Hosted

负责托管运行管理：

- 启动 hosted 来源
- 停止 hosted 来源
- 重启 hosted 来源
- 查看运行状态
- 查看最近日志

## 7. 页面设计要点

### Sources 页面

展示重点：

- 名称
- 类型
- 是否启用
- 连接状态
- 工具数量
- 是否托管
- 最后刷新时间

操作重点：

- 编辑
- 启停
- 刷新
- 查看工具
- 删除

### Tools 页面

这是核心页面，重点是“最后到底暴露了什么”。

展示重点：

- `exposedName`
- `originalName`
- `sourceName`
- `sourceKind`
- `enabled`
- `conflictStatus`

操作重点：

- 启用/禁用暴露
- 修改暴露名
- 恢复默认命名

### Hosted 页面

展示重点：

- 名称
- 来源类型
- 当前状态
- 自启动开关
- 最近启动时间
- 最近错误

操作重点：

- 启动
- 停止
- 重启
- 查看日志
- 编辑启动配置

## 8. 设计原则

### 结果优先

先让你看到：

- 接了哪些来源
- 暴露了哪些工具
- 托管服务是否正常

### 少页面

首版维持三页结构，不把它做成重后台。

### 少配置心智

常用设置直接展示，高级项折叠。

### 文件存储优先

自用场景下优先文件存储：

- 简单
- 易调试
- 可直接修复
- 便于配合 git 备份

## 9. 推荐存储形态

```text
data/
  sources/
    github.json
    notion.json
    local-dev.json
    hosted-fetcher.json
  exposures.json
  hosted-state.json
  logs/
```

## 10. MVP

首版必须完成：

- Source 列表
- 新增/编辑/删除 Source
- Source 能力刷新
- 工具汇总页
- 工具启用/禁用
- 工具重命名
- Hosted 服务启动/停止/重启
- Hosted 日志查看
- 统一 MCP 暴露入口
