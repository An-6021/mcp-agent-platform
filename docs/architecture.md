# Architecture

## 目标

新项目只解决一个核心问题：

让用户在 AI 工具里只配置一个本地 `stdio` MCP 命令，同时仍然可以通过远程配置做到个性化的服务编排。

## 核心原则

- 控制面和数据面分离
- 用户工具只连本地 agent
- 远程服务只下发配置，不承接全量调用
- 默认优先直连远程 MCP，上游必须经过中继时再单独扩展

## 一阶段模块

### `control-plane-api`

职责：

- 返回某个工作区的 MCP 配置
- 校验简单 Bearer Token
- 从本地数据目录读取工作区配置，作为 MVP 的持久化方式

### `agent`

职责：

- 以 `stdio` 启动，成为用户工具侧唯一 MCP 入口
- 拉取远程工作区配置
- 写入与读取本地缓存
- 将配置交给 `runtime`

### `runtime`

职责：

- 建立到上游 MCP 的连接
- 聚合 Tools / Resources / Prompts
- 通过前缀路由请求到目标上游

### `shared`

职责：

- 定义远程配置 Schema
- 定义缓存结构
- 定义名称前缀与资源 URI 映射

## 数据流

1. 用户在 AI 工具里配置一个本地命令，例如 `npx mcp-agent-platform ...`
2. 本地 agent 启动，携带工作区标识和令牌访问控制面
3. 控制面返回工作区配置
4. agent 将配置写入本地缓存
5. runtime 连接上游 MCP 并通过 `stdio` 暴露给工具
6. 后续工具请求由 runtime 直接路由到对应上游

## agent 启动参数

首版保留两种配置来源：

- `--base-url + --workspace`
- `--config-url + --workspace`

之所以仍然保留 `--workspace`，是因为本地缓存和日志都需要稳定的工作区键。

令牌支持两种方式：

- `--token`
- `--token-env`

默认更推荐 `--token-env`，避免把敏感信息直接写进工具配置。

## 首版风险

- 不同 AI 工具对 MCP 可选能力的支持存在差异
- 远程配置服务不可用时，缓存策略必须可靠
- `local-stdio` 上游仍依赖用户本机环境，需要在文档里清楚说明

## 后续演进

- 支持配置版本和增量刷新
- 支持更细粒度的用户或团队权限
- 为少数必须中继的服务增加 `relay-http` 类型
- 增加 Web 管理后台
